# エージェント設計パターン早見表（RASHIN 運用カタログ）

> このファイルは「必要なときだけ」参照する資料です（常時読み込みではないので、ふだんの動作は軽いまま）。
> RASHIN のメインエージェント（羅針）は、複雑・多段・品質重視の依頼を受けたとき、ここを見て
> 適したパターンを一言で提案してから進めます。
>
> 出典（一次情報）: Anthropic「Building Effective Agents」
> https://www.anthropic.com/engineering/building-effective-agents

---

## 使い方（羅針向け）

1. 依頼が単純なら、パターンを意識せず普通に処理する（カタログ参照は不要）。
2. 依頼が複雑・多段・品質重視なら、下の「判断リスト」で当てはまるパターンを選び、
   **「これは〇〇というやり方が向いています」と一言提案**してから動く。
3. パターンは会話だけで適用できる（スクリプト不要）。反復・大量・厳密化が必要なときだけ
   `Workflow` でコード化を検討する。

---

## 判断リスト（作業の特徴 → 推奨パターン）

| 作業の特徴 | 推奨パターン | ひとことで |
|---|---|---|
| きれいな手順に分解できる | **Prompt chaining** | 順番に処理し、各段でチェック |
| 依頼の種類がバラバラ | **Routing** | まず分類し、専門担当へ振り分け（RASHIN の既定動作） |
| 独立した小作業が複数／急ぎ／多数決したい | **Parallelization** | 並行実行（分担 or 投票）で速度・信頼性↑ |
| 分解の仕方が事前に読めない複雑作業 | **Orchestrator-workers** | 中央が動的に分解しワーカーへ委譲（RASHIN の土台） |
| 品質を詰めたい文章・コード・翻訳 | **Evaluator-optimizer** | 生成役⇄評価役のループで反復改善 |
| 道具を使って自分で考え動いてほしい | **Autonomous agent** | LLM が自律的に手順・ツールを判断 |

## 一次パターン（公式：Building Effective Agents）

すべての土台は **Augmented LLM**＝検索・ツール・メモリで拡張した LLM。以下はその組み合わせ方。

### Prompt chaining（プロンプト連鎖） — Workflow
- **何か**: タスクを順番のステップに分解し、前段の出力を次段へ。各段の間に合否ゲートを挟める。
- **向く**: 手順に分解でき、各段を単純化すると精度が上がるとき。
- **注意**: 段が増えるほど所要時間が伸びる。
- **RASHIN**: 例「論点出し → 調査 → 起草 → 校正」を段ごとに別エージェントへ。

### Routing（ルーティング） — Workflow
- **何か**: 入力を分類し、それぞれ最適化された専門処理へ振り分ける。
- **向く**: 依頼の種類がバラバラで、種類ごとに対応を分けた方が良いとき。
- **注意**: 入口の分類が正確であること（間違えると下流が全部ズレる）。
- **RASHIN**: 羅針が依頼を判定し各プリセットへ振る＝既定動作。

### Parallelization（並行化） — Workflow
- **何か**: タスクを並行実行する。(a) 分担（セクション分割）、(b) 投票（同じ問いを複数に解かせ多数決）。
- **向く**: 独立した小作業が複数あるとき、速度や信頼性を上げたいとき。
- **注意**: 分担の切り分け／投票の集約ルールを決めておく。
- **RASHIN**: 1メッセージで複数エージェントを同時起動。

### Orchestrator-workers（オーケストレーター＝ワーカー） — Workflow
- **何か**: 中央の LLM がタスクを動的に分解し、ワーカー LLM へ委譲、結果を統合する。
- **向く**: 必要なサブタスクが事前に読めない複雑な作業。
- **注意**: Parallelization と似るが、分解が「固定」でなく「動的」な点が違う。
- **RASHIN**: 羅針＝中央、サブエージェント＝ワーカー。RASHIN そのものの構造。

### Evaluator-optimizer（評価・改善ループ） — Workflow
- **何か**: 生成役が作り、評価役が採点・指摘し、それを受けて作り直す。基準を満たすまで反復。
- **向く**: 評価基準が明確で、反復すると目に見えて良くなるタスク（文章・コード・翻訳）。
- **注意**: 品質は上がるがコストと時間がかさむ。基準が曖昧だと効かない。
- **RASHIN**: 例「doc-writer 起草 → proofreader 評価 → 改稿」を数回。

### Autonomous agent（自律エージェント） — Agent
- **何か**: LLM が自分でプロセスとツール利用を方向づけ、環境からのフィードバックで進める。
- **向く**: 手順を事前に固定できず、探索的に進める必要があるとき。
- **注意**: 自由度が高いぶんコスト・暴走リスク管理が要る。停止条件とツール権限を絞る。
- **RASHIN**: サブエージェントは1タスク内で自律的に動く（ツール権限は定義で制限）。

## 補足パターン（二次情報・現場でよく使う派生）

公式の一次パターンではないが、実務で頻出。必要に応じて組み合わせる。

- **ReAct（推論＋行動）**: 「考える→道具を使う→観察する」を交互に繰り返す。Autonomous agent の典型的な動かし方。
- **Plan-and-Execute（計画してから実行）**: 先に全体計画を立て、その後に各ステップを実行。Prompt chaining ＋ Orchestrator の中間。
- **Reflection / Self-critique（自己批評）**: 1体が「書く→自分で批評→直す」。Evaluator-optimizer の単体版（軽量だが客観性は落ちる）。
- **Human-in-the-loop（人間の確認を挟む）**: 重要な分岐や外部送信の前にユーザー承認を入れる。RASHIN では既定の安全姿勢。

## 配役（キャスティング）のルール

合うエージェントが無いときの決め方:

1. **即席配役（確認なし）**: まず汎用エージェントにプロンプトで役を与えて動かす。事前確認は不要。
   - 役の組み立て4点: 専門性／観点・トーン／制約（字数・形式・禁止）／ツール（読む・書く・Web可か）。
2. **昇格（2〜3回で提案）**: 同じ役が 2〜3 回出てきたら、`my-agents/` への正式化（ファイル化）を提案する。
3. **安全側に倒す場面**: ファイル編集・外部送信・繰り返しの重要作業は、即席より
   **権限を絞った専用エージェント**を優先する。
